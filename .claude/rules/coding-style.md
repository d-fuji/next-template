# コーディングスタイル

## イミュータビリティ（CRITICAL）

常に新しいオブジェクトを生成する。既存オブジェクトを変更しない。

```typescript
// ✅ Good — スプレッド演算子で新オブジェクト
const updated = { ...user, name: "New Name" };
const filtered = items.filter((item) => item.active);
const mapped = items.map((item) => ({ ...item, processed: true }));

// ❌ Bad — ミューテーション
user.name = "New Name";
items.splice(0, 1);
items.push(newItem);
```

理由: イミュータブルなデータは副作用を防ぎ、デバッグを容易にする。

## ファイルサイズ

| 指標 | 目安 | 最大 |
|------|------|------|
| 1 ファイル | 200-400 行 | 800 行 |
| 1 関数 | 20-30 行 | 50 行 |
| ネスト深度 | 2-3 段 | 4 段 |

超過した場合はモジュール分割を検討する。

## エラーハンドリング

- すべてのレベルでエラーを明示的に処理する
- UI 向けコードはユーザーフレンドリーなメッセージを表示
- サーバー側は詳細なエラーコンテキストをログに記録
- エラーを黙って握りつぶさない（空の catch ブロック禁止）

## 入力バリデーション

システム境界（ユーザー入力、外部 API）では常にバリデーションする:

- Zod スキーマでバリデーション
- 明確なエラーメッセージで早期に失敗
- 外部データ（API レスポンス、ユーザー入力、ファイル内容）を信頼しない

## コード品質チェックリスト

作業完了前に確認:

- [ ] コードが読みやすく、適切に命名されている
- [ ] 関数が小さい（50 行以下）
- [ ] ファイルが集中している（800 行以下）
- [ ] 深いネストがない（4 段以下）
- [ ] 適切なエラーハンドリング
- [ ] ハードコードされた値がない（定数 or 設定を使用）
- [ ] ミューテーションがない（イミュータブルパターン使用）
- [ ] console.log が残っていない
