# セキュリティガイドライン

## コミット前の必須チェック

- [ ] ハードコードされたシークレットがないこと（API キー、パスワード、トークン）
- [ ] すべてのユーザー入力がバリデーションされていること
- [ ] SQL インジェクション対策（パラメータ化クエリ / Prisma 使用）
- [ ] XSS 対策（HTML のサニタイズ）
- [ ] 認証・認可が正しく設定されていること
- [ ] エラーメッセージが機密情報を漏洩しないこと

## シークレット管理

- シークレットをソースコードにハードコードしない
- 常に環境変数またはシークレットマネージャーを使う
- 必要なシークレットが起動時に存在することをバリデーションする
- 漏洩した可能性のあるシークレットはローテーションする

## API エンドポイント

- すべての保護ルートで `.withAuth()` を使用する
- リクエストボディ/クエリは Zod スキーマでバリデーションする
- 内部エラーの詳細をクライアントに返さない
- レート制限を検討する

## パスワード

- プレーンテキストで比較しない（`bcrypt.compare()` を使用）
- パスワードをログに出力しない
- パスワードポリシーを実装する

## セキュリティ問題発見時の対応

1. 即座に作業を停止
2. **security-reviewer** エージェントを使用
3. CRITICAL な問題を修正してから続行
4. 漏洩した可能性のあるシークレットをローテーション
5. 類似の問題がないかコードベース全体をレビュー
